# 基于大模型的测试场景生成与智能测试工具

## 简介

- 随着 Web Agent 快速发展，人们对其在复杂 Web 应用中的测试可用性提出了更高要求。
- 现有 Web Agent 在 WebArena 等基准上仍表现出较高的任务失败率，原因在于：
  1. Web 应用业务逻辑复杂且高度领域化。
  2. 大模型存在幻觉，且缺乏待测应用的领域知识，容易输出与事实不符的信息。
  3. 难以在执行完操作后准确判定任务是否成功。
- 软件文档（如用户手册）蕴含大量业务知识，可帮助 Agent 更好地理解功能、判断执行结果。
- 本课程实践旨在设计一种基于大模型的测试场景生成与自主执行工具：
  1. 自动提取用户手册中的知识，生成功能覆盖全面、粒度适宜、具有可执行性的测试场景。
  2. 基于大模型智能体，自主执行测试场景，对目标应用开展全面的功能测试。

## 任务

### 任务一：基于用户手册的测试场景自动生成

- 目标：基于大模型检索增强技术（RAG），从用户手册中提取知识，生成测试场景。
- 要求：
  - 能够根据用户手册提取软件的主要功能点。
  - 根据每个功能点能够生成结构化的测试场景。
  - 支持功能点、测试场景的可视化展示。
- 测试场景：
  - 构成：`[ [step]+ [expectation]? ]+`
  - 操作步骤：完成功能所需的具体操作步骤。
  - 预期状态：功能完成后的预期状态，相当于测试预言，包含用于评估功能是否成功完成的关键特征。

### 任务二：测试场景驱动的智能测试智能体

- 目标：构建基于大模型的 Web 测试智能体，自动执行任务一生成的测试场景。
- 架构需至少包含以下要素（可自主扩展）：
  - 规划：根据测试场景自主规划执行计划。
  - 记忆：辅助执行的上下文信息。
  - 执行：与浏览器交互，执行规划动作。
  - 验证：根据整个测试轨迹验证测试场景是否执行成功，该功能是否正确。

## 评分标准

### 基础功能档

- 任务一：根据用户手册能够识别主要功能点，并生成主要功能的测试场景，测试场景的生成格式符合要求。
- 任务二：智能体能够执行任务一中识别的简单测试场景，能够验证执行完整性与功能正确性。

### 提升创新档

- 任务一：
  - 生成测试场景的正确性与全面性：确保从用户手册等文档中检索到的信息真实有效，尽量避免大模型“幻觉”导致的错误；尽可能全面地覆盖文档中包含的功能并避免遗漏。
  - 生成测试场景的粒度与可执行性：功能划分的粒度应合适，既不能过细导致难以组合成实际功能，也不宜过粗以致无法指导具体操作；同时保证生成的用户故事具有可执行性，即能够被现有的 Web 测试智能体有效执行。
- 任务二：
  - 测试场景执行的通过率与稳定性：提升智能体执行测试场景的准确率，对于简单、中等难度测试场景能够正确通过，甚至通过部分困难场景；此外，也可考虑提升智能体执行的效率。
  - 支持对测试场景进行变异，能够识别典型的应用错误（包括：执行异常、布局问题、语义错误等）。
  - 测试场景验证：能够根据执行轨迹正确识别执行结果（成功与失败），对于执行失败准确输出失败原因。
- 鼓励所有的创新方法设计以及评估策略设计（评估方法与指标）等等。

## 补充

- 目标 Web 应用为 4gaboards：
  - 用户手册地址：https://docs.4gaboards.com/
  - 应用 demo 地址：https://demo.4gaboards.com/
- 计算设备：对于个人计算设备的计算能力无特殊要求，能完成赛题即可。
- 软件依赖：编程语言及所用技术不限；需使用版本控制工具（Git 等）。
- 大模型：国产大模型 GLM-4.6V/4.7、Qwen3-VL-Plus、DeepSeek V4 Pro。当前项目实现以 `PLANv2.md` 为准，默认文本模型为 DeepSeek V4 Pro（`deepseek-v4-pro`），并保留 Browser Use 托管 LLM 作为显式可选替代方案。
